## 一、产品概述

餐饮AI店长是一套面向中小餐饮店的AI经营管理系统，以"四大智能体 + 25位AI数字员工"为核心形态，覆盖从选址开店到复购运营的完整生命周期。

产品的特殊之处在于：**它不是一套从PPT出发的理论系统，而是从一家真实运营的餐饮店——"哇塞猛火猪油炒饭"——中生长出来的实战工具。** 每一个功能模块都经历了真实门店场景的检验，每一个数据表都承载过真实的经营数据。

当前系统已构建完成30个前端页面、24张后端数据表、4个MCP工具，形成了从数据采集到决策辅助再到执行反馈的完整闭环。

---

## 二、产品理念

### 1. 实战验证不是PPT

每个模块都在自己店里测试过。不是"理论上可行"，而是"今天就在用"。成本核算跑过真实BOM，差评处理回复过真实差评，库存预警触发过真实补货。这是系统与纯方案型产品的根本区别。

### 2. 全链路覆盖

从选址分析到复购引擎，覆盖餐饮经营的完整生命周期。不是单点工具的堆砌，而是一条贯穿"开店→运营→增长"的完整链路，每个环节都有对应的功能模块和AI数字员工。

### 3. 纯增量架构

不重构、只增量。V1→V2→V3的每次迭代都是向上叠加，底层表结构不改动、前端页面不推翻。系统始终保持干净、可扩展，新功能接入像插拔模块一样简单。

### 4. 数据飞轮

OKF餐饮知识库提供行业基准 → 门店数据回流形成个体画像 → 多店对比发现差异化机会 → 知识库持续优化。数据不是静态存储，而是驱动系统越用越聪明的飞轮。

### 5. 数据主权归用户

用户完全自持数据，支持私有化部署。数据存放在用户自己的Supabase实例中，不经过任何第三方中转。企业定制版支持专属部署，数据不出用户掌控范围。

## 三、演进历程

### V1 · 基础层（6张表）

|--------|------|

| `purchase_acceptance` | 采购验收记录 |
| `supplier_scores` | 供应商评分 |
| `ingredient_inventory` | 食材库存管理 |
| `food_safety_checklist` | 食品安全检查清单 |
| `equipment_registry` | 设备登记台账 |
| `maintenance_logs` | 设备维护日志 |
| `data_quality_metrics` | 数据质量指标 |
| `decision_confidence_logs` | 决策置信度日志 |
| `loss_records` | 损耗记录 |
| `loss_daily_summary` | 损耗日报汇总 |
| `staff_profiles` | 员工档案 |
| `staff_schedule` | 排班管理 |

| 数据表 | 用途 |
|--------|------|

## 六、功能全览

除四大智能体核心模块外，系统还包含以下独立功能模块：

### 选址分析（`site.html`）

输入目标区域，输出3km商圈画像（人口结构、消费能力、交通配套）+ 竞品密度热力图 + 客群分析 + 选址综合评分。帮助创业者在开店前做出数据驱动的选址决策。

### 开业流程（`opening.html`）

从证照办理到设备采购的全流程指引，覆盖营业执照、食品经营许可证、消防安全、环保审批、设备清单、装修验收等关键节点，每一步都有材料清单和注意事项。

### 品牌设计（`brand.html`）

提供品牌命名方案、Logo设计方向、配色方案、门头排版建议和制作尺寸参考。不依赖外部设计工具，直接在系统内完成品牌视觉的基础搭建。

### 数据治理（`data-governance.html`）

对系统内的数据进行置信度评估，标记数据质量指标，帮助用户识别哪些数据可信、哪些需要补充。配合V3迭代层的`data_quality_metrics`表，实现数据质量的持续监控。

### 免费诊断（`diagnosis.html`）

引流入口产品。内置13个餐饮品类模板（炒饭、奶茶、汉堡、火锅等），支持按位置差异化分析。每日3次免费使用，用户无需注册即可体验核心价值，降低尝试门槛。

### 多店管理（`multi-store.html`）

多门店统一看板，支持跨门店数据对比、统一配置下发、分门店权限管理。适用于拥有2家以上门店的经营者。

### 连锁后台（`chain-admin.html`）

连锁品牌统一管理后台，支持品牌级菜单统一下发、分门店差异化配置、总部数据汇总、加盟商权限分级。适用于连锁品牌总部。

### 仪表盘（`dashboard.html`）

经营数据总览页面，一屏展示关键经营指标（日营收、订单量、客单价、毛利、差评率等），支持按日/周/月切换查看趋势。

### 智能体聊天（`agent-chat.html`）

统一聊天界面，用户可通过自然语言与四大智能体交互。支持3种主题配色（金色/蓝色/紫色），根据当前对话的智能体自动切换。

### 客服Bot（`xiaowa-bot.html`）

智能小哇的前端载体，内置7条高频FAQ、实时菜品查询能力、一键转人工功能。作为系统的统一入口和帮助中心。

```
┌─────────────────────────────────────────────────────┐
│                    用户层（前端）                      │

│    opening / brand / data-governance / diagnosis /    │
│    multi-store / chain-admin / dashboard /            │
│    agent-chat / portal / index ...                    │
├─────────────────────────────────────────────────────┤
│                    AI 能力层                          │
│              Coze（扣子）智能体平台                     │
│         4大智能体 · 25位AI数字员工                      │
│           OKF 餐饮知识库                               │
├─────────────────────────────────────────────────────┤
│                   MCP 服务层                          │
│        Python + FastAPI + SSE · 端口8765              │
│     get_menu / get_inventory_status /                 │
│     get_store_info / get_daily_summary                │
├─────────────────────────────────────────────────────┤
│                   数据层（后端）                        │
│              Supabase（PostgreSQL）                   │
│            24张数据表 · V1/V2/V3三层                   │
│        保活机制：每3天自动ping                          │
└─────────────────────────────────────────────────────┘
```

### 前端

- **托管方式**：GitHub Pages 静态托管，零服务器成本
- **页面数量**：30个HTML页面
- **设计理念**：苹果设计哲学——1页1功能，≤3个输入，一键出结果，零术语，首次打开为空
- **技术特点**：纯静态HTML，无构建依赖，加载快、维护简单

### 后端

- **数据库**：Supabase（基于PostgreSQL）
- **数据表**：24张，按V1/V2/V3三层组织
- **保活机制**：Supabase免费层有自动暂停策略，系统每3天自动ping一次保持活跃
- **数据安全**：用户数据完全自持，支持私有化部署

### AI平台

- **平台选择**：Coze（扣子）
- **智能体编排**：4大智能体 + 25位数字员工
- **知识库**：OKF餐饮知识库，提供行业基准和领域知识

## 九、MCP Server

### 概述

MCP（Model Context Protocol）Server 是连接AI智能体与门店数据的核心桥梁。基于Python + FastAPI + SSE（Server-Sent Events）构建，运行在8765端口，为AI智能体提供实时的门店数据查询能力。

### 技术栈

| 组件 | 技术选型 |
|------|---------|
| 语言 | Python |
| Web框架 | FastAPI |
| 通信协议 | SSE（Server-Sent Events） |
| 端口 | 8765 |
| 数据源 | Supabase（PostgreSQL） |

### 四个核心工具

#### 1. `get_menu` — 查询门店菜品

查询指定门店的菜品信息，返回菜品名称、价格、分类和上下架状态。供AI智能体在点餐推荐、菜品分析、菜单优化等场景中调用。

**典型场景**：顾客询问"你们有什么炒饭？"、老板询问"哪个菜品毛利率最低？"

#### 2. `get_inventory_status` — 查询库存状态

查询指定门店的食材库存状态，支持低库存预警。当某项食材库存低于安全阈值时，主动标记并提醒。

**典型场景**：店长询问"今天需要补什么货？"、系统自动触发补货提醒。

#### 3. `get_store_info` — 查询门店信息

查询门店的基础信息，包括门店名称、地址、营业时间、经营品类等。供AI智能体在客服对话、门店对比等场景中调用。

**典型场景**：顾客询问"你们店几点开门？"、竞品分析时获取自身门店定位。

#### 4. `get_daily_summary` — 查询日报数据

查询指定门店的日报汇总数据，包括日营收、订单量、客单价、毛利等关键经营指标。供AI智能体在财务分析、经营看板、决策辅助等场景中调用。

**典型场景**：老板询问"今天赚了多少？"、系统自动生成日报推送。

### MCP的价值

MCP Server的建成意味着餐饮AI店长已经具备了与瑞幸等头部品牌同等的AI-数据连接能力。AI智能体不再是"只会聊天"的对话机器人，而是能够实时读取门店数据、基于真实数据给出建议的经营助手。

## 十三、核心链路与设计理念

### 核心链路

```
感知层                    思考层                    表达层                    可视化层
┌──────────────┐     ┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│ 竞品监控      │     │ 雷军决策模型  │     │ 去AI味改写    │     │ SVG图表      │
│ 数据采集      │ ──→ │ 成本分析      │ ──→ │ 营销文案      │ ──→ │ 财务报表      │
│ 用户反馈      │     │ 定价策略      │     │ 差评回复      │     │ 经营看板      │
│ 库存感知      │     | 菜品分析      │     | 触达话术      │     | 数据治理      │
└──────────────┘     └──────────────┘     └──────────────┘     └──────────────┘
```

**感知**：系统持续采集竞品动态、门店数据、用户反馈、库存状态等输入信号。

**思考**：基于雷军决策模型和OKF知识库，对感知到的信息进行成本分析、定价推演、菜品评估等多维度思考。

**表达**：将思考结果转化为用户可理解的输出——去AI味的营销文案、专业的差评回复、精准的触达话术。

**可视化**：最终通过SVG图表、财务报表、经营看板等形式呈现，让数据一目了然。

### 设计理念

遵循**苹果设计哲学**：

| 原则 | 含义 | 体现 |
|------|------|------|
| 1页1功能 | 每个HTML页面只解决一个问题 | 30个页面各自独立，不混杂功能 |
| ≤3个输入 | 用户操作步骤不超过3步 | 成本核算只需输入菜品原料和采购价 |
| 一键出结果 | 点击一次即生成完整结果 | 差评处理一键生成3版回复 |
| 零术语 | 不使用专业术语，用大白话 | "今天赚了多少"而非"日营收汇总" |
| 首次打开为空 | 不预填假数据，尊重用户 | 新用户打开看到的是空白引导页 |

从选址分析到复购引擎，覆盖餐饮经营完整生命周期。25+功能模块、25位AI数字员工，是目前餐饮AI领域覆盖场景最完整的产品。竞品中，客如云覆盖5个场景，美团覆盖部分场景，肯德基/瑞幸仅覆盖点餐场景，茶百道仅覆盖单场景。

### 3. 数据资产意识超前 — OKF知识库 + 数据飞轮

在大多数餐饮AI产品还停留在"工具"层面时，餐饮AI店长已经构建了OKF知识库 + 数据飞轮的数据资产体系。数据不是用完即弃的消耗品，而是持续增值的资产——越用越聪明，越用越精准。

### 4. MCP能力已就绪 — 与瑞幸同一起跑线

MCP Server的建成意味着餐饮AI店长已经具备了与瑞幸等头部品牌同等的AI-数据连接能力。AI智能体可以实时读取门店数据，基于真实数据给出建议。这一能力在中小餐饮AI领域属于领先水平。

### 5. 纯增量架构 — 干净可扩展

V1→V2→V3的纯增量架构，使系统始终保持干净和可扩展。新功能接入不需要重构旧代码，新表添加不需要修改旧表结构。这种架构的健康度在快速迭代的产品中极为罕见。

## 十五、五大待补短板

### 1. 用户基础为零 — 缺获客渠道

当前仅1家自有门店在用，缺乏外部用户和获客渠道。这是最大的短板——产品再好，没有人用就没有飞轮。需要建立从免费诊断到付费转化的完整漏斗。

### 2. 生态未接入 — 千问/微信AI待执行

四项生态接入（微信AI、千问Agent、MCP公网、腾讯云混元）均已完成调研但尚未执行落地。生态接入是获取用户基础的关键路径，需要尽快推进。

### 3. 品牌认知为零 — 需从零建立信任

餐饮AI店长作为一个新品牌，市场认知度为零。中小餐饮店主对AI工具的信任度本就不高，从零建立品牌认知和用户信任是一项长期工作。

### 4. 数据飞轮待启动 — 数据量不足

数据飞轮的框架已经就绪，但只有1家门店的数据，飞轮还转不起来。需要更多门店接入，形成足够的数据量和多样性，飞轮才能真正加速。

### 5. 内容生成能力有限 — 缺"从零生成图片/设计"

当前系统的内容生成能力偏重文本（文案改写、差评回复），缺乏从零生成图片和设计的能力。在餐饮营销场景中，视觉内容（菜品图、海报、菜单排版图）的需求量很大，这是需要补齐的能力。
