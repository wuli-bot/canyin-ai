## 2. Coze（扣子）— 主力AI平台

### 2.1 依赖概述

| 项目 | 内容 |
|------|------|
| **角色定位** | 主力AI平台，承载全部AI能力 |
| **当前规模** | 4大智能体 + 25位AI数字员工 |
| **核心功能** | AI对话、客服Bot、日报生成、差评处理、菜品分析、选址评估等 |
| **影响范围** | 所有AI对话、客服Bot、日报生成、差评处理等核心功能 |
| **不可用后果** | 系统全部AI功能瘫痪，无法服务终端用户 |

### 2.2 风险识别

| 风险编号 | 风险事件 | 风险描述 | 触发条件 |
|----------|----------|----------|----------|
| R-CZ-01 | API限流 | 高峰期请求超过Coze限流阈值，请求被拒绝或排队 | 并发请求量突增 / 套餐配额耗尽 |
| R-CZ-02 | 服务中断 | Coze平台整体或区域性服务不可用 | 平台故障 / 维护 / 云服务商问题 |
| R-CZ-03 | API接口变更 | Coze更新API版本，现有接口不兼容 | 平台版本升级 / 接口废弃 |
| R-CZ-04 | 计费政策调整 | Coze调整计费规则，导致成本大幅上升或配额缩减 | 平台商业策略变更 |
| R-CZ-05 | 平台封禁 | 账号因违规操作或误判被封禁 | 内容审核触发 / 账号异常 |
| R-CZ-06 | 响应延迟 | Coze响应时间显著增加，影响用户体验 | 平台负载过高 / 网络问题 |

### 2.3 影响评估

| 影响维度 | 评估 |
|----------|------|
| **功能影响** | 全部AI对话、客服Bot、日报生成、差评处理、12项餐饮技能全部不可用 |
| **用户影响** | 终端用户无法获取AI服务，客服无人响应 |
| **业务影响** | 外卖差评无法处理（差评预警失效）、日报无法生成、经营决策无AI辅助 |
| **数据影响** | 无直接数据丢失风险（数据存储在Supabase） |
| **恢复难度** | 需切换至千问Agent备用平台（4-6周完整部署，紧急切换约30分钟） |

### 2.4 降级方案

#### 短期方案（即时 ~ 24小时）

| 措施 | 说明 | 预期效果 |
|------|------|----------|
| 触发千问Agent紧急切换 | 备用路由服务检测到Coze不可用后，自动将AI请求转发至千问Agent | 核心AI对话功能恢复（依赖千问Agent已完成基础部署） |
| 启用H5静态客服 | xiaowa-bot.html 独立运行，提供基础FAQ应答 | 保证基础客服能力不中断 |
| 人工介入差评处理 | 差评预警改为人工查看+手动回复 | 差评处理不中断但效率降低 |
| 飞书群通知 | 自动推送告警至运维飞书群，通知相关人员 | 确保团队第一时间知情 |

#### 中期方案（1 ~ 7天）

| 措施 | 说明 | 预期效果 |
|------|------|----------|
| 千问Agent功能补齐 | 在备用平台上补齐4大智能体的核心能力映射 | AI功能基本恢复至80%水平 |
| 评估Coze恢复可能性 | 联系Coze客服/查看官方公告，评估恢复时间 | 明确是否需要长期切换 |
| 用户公告 | 通过小程序/H5发布服务降级通知 | 管理用户预期，减少投诉 |
| 临时扩容千问Agent配额 | 根据实际流量调整千问API调用配额 | 确保备用平台稳定承载 |

#### 长期方案（1周以上）

| 措施 | 说明 | 预期效果 |
|------|------|----------|
| 双平台并行架构 | Coze + 千问Agent双活部署，按权重分流 | 消除单点依赖，任一平台故障自动切换 |
| 多平台AI能力抽象层 | 统一AI接口层，支持Coze/千问/OpenAI等多后端 | 平台切换对上层透明，降低迁移成本 |
| Coze商务洽谈 | 与Coze官方建立企业级合作，获取SLA保障 | 降低限流/封禁风险 |
| 成本优化模型 | 建立多平台成本对比模型，动态选择最优平台 | 控制整体AI调用成本 |

### 2.5 触发条件与判断逻辑

```
触发降级的条件（满足任一即触发）：
┌─────────────────────────────────────────────────────────┐
│ 1. Coze API连续 3 次请求返回 5xx 错误                    │
│ 2. Coze API连续 5 次请求超时（>30秒）                    │
│ 3. Coze API返回 429（限流）且 5 分钟内未恢复              │
│ 4. Coze 平台官方公告确认服务中断                          │
│ 5. 账号被封禁（返回 401/403 且确认非密钥问题）             │
└─────────────────────────────────────────────────────────┘

恢复回切条件（全部满足才回切）：
┌─────────────────────────────────────────────────────────┐
│ 1. Coze API连续 10 分钟正常响应（成功率 > 99%）           │
│ 2. Coze 官方确认服务已恢复                                │
│ 3. 当前非业务高峰期（避开 11:00-13:00, 17:00-20:00）      │
│ 4. 灰度回切验证通过（先切 10% 流量，观察 15 分钟）        │
└─────────────────────────────────────────────────────────┘
```

| 风险编号 | 风险事件 | 风险描述 | 触发条件 |
|----------|----------|----------|----------|
| R-QW-01 | 切换失败 | 紧急切换时千问Agent未完成部署或配置有误 | 备用平台未完成上线 / 配置过期 |

```
备用路由服务决策流程：
═══════════════════════════════════════════

[接收AI请求]
       │
       ▼

       │
       ▼

  │
       │
       ▼
[切换执行]
  ├── 主→备切换：立即执行，发送告警

  │
       │
       ▼
[请求转发]
  ├── Coze路由：原样转发，附加trace_id
  ├── 千问路由：转换请求格式，映射工具调用，附加trace_id
  └── 响应统一格式化后返回调用方
```

### 3.5 降级方案

#### 短期方案（千问Agent尚未完成部署时）

| 措施 | 说明 |
|------|------|
| H5静态FAQ | xiaowa-bot.html 提供基础关键词匹配问答 |
| 人工客服兜底 | 运维人员通过飞书群手动响应用户咨询 |
| 降级公告 | 小程序/H5发布"AI服务暂时降级"通知 |

#### 中期方案（千问Agent基础部署完成）

| 措施 | 说明 |
|------|------|
| 紧急切换至千问Agent | 路由服务自动切换，恢复核心AI对话 |
| 限制非核心功能 | 暂时关闭选址分析、设备选型等P1功能，优先保障P0 |
| 动态成本控制 | 设置千问API日消费上限，超限自动降级为静态FAQ |

#### 长期方案

| 措施 | 说明 |
|------|------|
| 多模型路由策略 | 按任务类型选择最优模型（简单任务→qwen-turbo，复杂任务→qwen-plus） |
| 本地模型探索 | 评估轻量级本地模型（如Qwen-7B本地部署）处理简单任务 |
| 多平台冗余 | 评估接入第三家AI平台（如智谱GLM），形成三重保障 |

### 3.6 触发条件

| 触发场景 | 触发条件 | 执行动作 |
|----------|----------|----------|

## 7. 应急响应流程

### 7.1 应急响应组织

| 角色 | 职责 | 响应时效 |
|------|------|----------|
| 运维负责人 | 整体协调、决策切换/回切 | P0: 5分钟内响应 |
| AI工程师 | Coze/千问平台技术处理 | P0: 10分钟内响应 |
| 数据库管理员 | Supabase故障处理 | P0: 10分钟内响应 |
| 前端开发者 | 小程序/H5切换处理 | P0: 15分钟内响应 |
| 业务负责人 | 用户沟通、公告发布 | P0: 15分钟内响应 |

### 7.2 应急响应时间线

```
T+0min    │ 检测到异常 / 收到告警
T+1min    │ 告警推送至飞书群
T+3min    │ 自动降级措施触发（如配置了自动切换）
T+5min    │ 运维负责人确认告警，启动应急响应
T+10min   │ 技术人员介入，评估影响范围
T+15min   │ 执行降级方案（如未自动触发）
T+30min   │ 验证降级方案效果，通知用户
T+60min   │ 制定恢复计划，预估恢复时间
T+4h      │ 执行恢复（如依赖已恢复）
T+24h     │ 归档事件报告，更新预案
```

### 7.3 联动降级策略

当多个依赖同时故障时，按以下优先级处理：

| 优先级 | 故障组合 | 处理策略 |
|--------|----------|----------|
| 最高 | Coze + Supabase 同时故障 | 优先恢复Supabase（数据安全>AI功能）→ 千问Agent + 本地缓存 |
| 高 | Coze + 微信小程序 同时故障 | 切换千问Agent + 启用H5入口 → 双路并行恢复 |
| 高 | Supabase + 微信小程序 同时故障 | 优先恢复Supabase → H5入口承载服务 |
| 中 | Coze + 千问Agent 同时故障 | 启用静态FAQ + 人工客服 → 紧急联系Coze/阿里云客服 |
| 低 | 全部依赖故障 | 静态H5 + 人工模式 → 逐个恢复（Supabase→Coze→小程序→千问） |
