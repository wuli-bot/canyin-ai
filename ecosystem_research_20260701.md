# 餐饮AI生态对外接入调研报告

完成日期：2026-07-01

---

## 摘要

本报告覆盖三个P1调研任务：MCP Server对外开放方案、千问Agent接入调研、微信AI生态接入调研。核心发现如下：

**MCP Server对外开放**：华为云、腾讯云TKE、阿里云均已提供MCP Server云端托管方案，推荐**腾讯云TKE容器部署**作为首选方案（与微信生态同源、中文文档完善），月成本约187元起步。参考瑞幸咖啡MCP开放平台做法，应采用Skill+MCP+CLI三通道开放、Bearer Token认证、免费开放核心能力的策略。

**千问Agent接入**：千问App于2026年6月3日全面开放第三方Agent/Skill生态，肯德基成为首个餐饮品牌接入方，通过Skill形式实现全国1.3万+门店AI点餐。技术对接基于阿里云百炼平台OpenAI兼容API，接入成本极低。当前Skill服务已上线，Agent形态仍在测试中。

**微信AI生态接入**：2026年6月8日微信正式发布开发者接入指引，提供自动模式（零代码）和开发模式（MCP协议深度定制）双通道。餐饮小程序应优先选择开发模式，基于mcp.json定义点单、查单等原子接口。AI专属卡已上线，为AI支付提供资金隔离安全方案。

**推荐执行路线图**：微信AI生态接入（P0/2周内启动）→ 千问Agent接入（P1/3周内启动）→ MCP Server公网部署（P2/4周内启动）。微信生态因14亿用户基础和餐饮小程序天然适配性，优先级最高。

## 第一章：MCP Server 对外开放方案

### 1.1 公网部署方案

#### 1.1.1 云平台推荐

当前国内三大云厂商均提供MCP Server托管方案，对比如下：

| 维度 | 腾讯云TKE | 华为云Flexus | 阿里云ECS/FC |
|------|-----------|-------------|-------------|
| MCP文档 | 完整托管指南，Codebuddy验证闭环 [(腾讯云文档)](https://cloud.tencent.com/document/product/457/124005) | 一键部署方案，含tavily/quickchart等示例 [(华为云文档)](https://support.huaweicloud.com/mcpsse-aislt/aislt-mcpsse.pdf) | Bedrock AgentCore Runtime（海外）/ECS自部署 [(AWS博客)](https://aws.amazon.com/cn/blogs/china/agentic-ai-infrastructure-practice-experience-series-four-mcp-server-from-local/) |
| 月成本（2核4G） | ~187元（按需） | ~187元（按需） | ~200元+ |
| 容器化支持 | 原生TKE+CCR镜像服务 | Flexus云服务器 | ACS/FC函数计算 |
| 与微信生态兼容 | 同源腾讯体系，内网互通 | 独立体系 | 独立体系 |
| 中文文档质量 | ★★★★★ | ★★★★ | ★★★ |

**推荐方案：腾讯云TKE容器服务部署**

理由：①与微信生态同属腾讯体系，未来对接微信AI生态内网延迟最低；②MCP Server托管文档最完善，含从镜像打包到Codebuddy验证的完整闭环；③TKE支持Streamable HTTP传输协议原生适配 [(腾讯云文档)](https://cloud.tencent.com/document/product/457/124005)。

#### 1.1.2 部署架构

```
用户/Agent → HTTPS(CDN/WAF) → TKE Ingress → MCP Server Pod(FastAPI)
                                              ↓
                                         业务数据库/缓存
```

具体步骤：
1. **镜像打包**：将现有Python/FastAPI MCP Server打包为Docker镜像，上传至腾讯云CCR
2. **TKE部署**：创建Deployment+Service，暴露Streamable HTTP端点（默认8000端口）
3. **公网暴露**：通过TKE Service的LoadBalancer类型获取公网IP
4. **域名绑定**：配置自定义域名指向LB地址
5. **HTTPS配置**：通过腾讯云SSL证书服务申请免费证书，绑定到Ingress

#### 1.1.3 域名与HTTPS配置

| 配置项 | 建议 |
|--------|------|
| 域名 | `mcp.canyin-ai.com`（主域名）或 `api.canyin-ai.com` |
| SSL证书 | 腾讯云免费DV证书（1年有效期）或付费OV证书 |
| HTTPS强制 | Ingress配置`tls`段，HTTP→HTTPS 301重定向 |
| CORS | 允许`*.canyin-ai.com`及已知AI客户端域 |

#### 1.1.4 安全策略

基于企业级MCP最佳实践 [(腾讯云开发者社区)](https://cloud.tencent.com.cn/developer/article/2695242)，建议采用**控制面/数据面分离**架构：

1. **网络层**：VPC隔离+安全组白名单，仅开放443(HTTPS)和健康检查端口
2. **认证层**：OAuth 2.0或Bearer Token认证（详见1.3节）
3. **限流层**：按租户/IP维度的请求限流（详见1.3节）
4. **审计层**：所有tool调用日志留存≥6个月，记录调用者、时间、参数、结果
5. **隔离层**：不同会话的缓存/凭据/变量完全分离
6. **注入防护**：对模型生成的参数做二次校验，关键操作需人工确认钩子

### 1.3 接口规范设计建议

#### 1.3.1 认证方式

参考瑞幸MCP采用Bearer Token方案 [(技术栈)](https://jishuzhan.net/article/2064622360598228994) 及阿里云OpenAPI MCP Server的OAuth双模式 [(阿里云文档)](https://help.aliyun.com/zh/openapi/user-guide/openapi-mcp-server-guide)，建议**双模式认证**：

| 模式 | 适用场景 | 实现方式 |
|------|---------|---------|
| **Bearer Token**（主推） | 个人开发者、AI客户端接入 | 手机号登录→获取Token→Header携带`Authorization: Bearer <token>` |
| **OAuth 2.0**（企业级） | 企业系统集成、CI/CD流水线 | 授权码+PKCE流程，短期Access Token+Refresh Token |

Token管理规范：
- 有效期：30天（与瑞幸一致）
- Token与用户会话绑定，严禁泄露或分享
- 支持Token刷新机制，过期前7天提醒
- 关键操作（修改菜单、确认订单）需二次确认

#### 1.3.2 限流策略

| 维度 | 限流规则 | 说明 |
|------|---------|------|
| 全局QPS | 100 QPS | 单实例承载上限 |
| 单Token | 30 RPM/300 RPD | 每分钟/每天请求限制 |
| 单工具 | 按工具分级 | 只读工具（get_menu等）：50RPM；写操作：10RPM |
| 突发 | 令牌桶算法 | 允许短期突发，平滑限流 |

错误响应格式：
```json
{
  "error": {
    "code": 429,
    "message": "Rate limit exceeded",
    "retry_after": 30
  }
}
```

#### 1.3.3 版本管理

采用语义化版本（SemVer）+ URL路径版本双轨制：

- **API路径版本**：`/v1/tools/get_menu`，当前v1，大版本变更时新增v2
- **MCP协议版本**：`protocolVersion`在初始化握手时协商
- **变更策略**：
  - PATCH：工具内部逻辑优化，无破坏性变更，兼容升级
  - MINOR：新增工具/可选参数，向后兼容
  - MAJOR：删除工具/修改必选参数/变更响应格式，需迁移指引

### 1.4 参考瑞幸咖啡MCP开放平台的差异化建议

#### 1.4.1 瑞幸MCP开放平台做法拆解

瑞幸咖啡于2026年6月9日上线AI开放平台 [(瑞幸微博)](https://m.weibo.cn/detail/5305753204166624)，提供**Skill+MCP+CLI三通道**接入方式：

| 接入方式 | 目标用户 | 核心体验 |
|---------|---------|---------|
| Skill | 普通用户 | 自然语言全自动点单，AI匹配门店+优惠券 |
| MCP | 开发者/AI客户端 | 标准MCP协议，对接豆包/Claude/Cursor等 |
| CLI | 极客开发者 | 终端一行命令安装部署，3秒完成 |

MCP端点：`https://gwmcp.lkcoffee.com/order/user/mcp`（StreamableHTTP）[(技术栈)](https://jishuzhan.net/article/2064622360598228994)

暴露8个工具：queryShopList、searchProductForMcp、queryProductDetailInfo、previewOrder、createOrder、queryOrderDetailInfo、cancelOrder、switchProduct。

**瑞幸的不足**（实测反馈）：
- `switchProduct`接口完全不可用（返回非法参数）
- 配送功能不生效（仅支持到店自取，文档与实际不一致）
- Java MCP SDK兼容性问题（旧版反序列化失败）
- 缺少地址管理、购物车等工具

#### 1.4.2 我们的差异化建议

| 差异化维度 | 瑞幸做法 | 我们的建议 |
|-----------|---------|-----------|
| **用户定位** | 面向C端消费者点单 | **面向B端餐饮店长运营管理**——查菜单、看库存、管门店、看数据 |
| **开放策略** | 单一消费场景（点咖啡） | **管理+消费双通道**——4个基础工具先覆盖管理场景，后续扩展消费下单 |
| **工具完整性** | switchProduct不可用、配送不生效 | **承诺工具100%可用**——每个上线工具必须通过完整E2E测试，不发布半成品接口 |
| **SDK兼容性** | Java SDK旧版导致握手失败 | **Python MCP SDK优先**（与FastAPI同栈），同时提供OpenAI Function Calling兼容模式 |
| **认证体验** | 手机号登录+30天Token | **手机号/微信双通道登录**，Token+OAuth双模式，企业客户支持API Key |
| **文档诚实性** | 文档列了配送但实际不支持 | **文档即契约**——只文档化已验证可用的能力，未上线功能标注为[Roadmap] |
| **三通道策略** | Skill+MCP+CLI | **Skill+MCP+CLI+Dashboard**——额外提供Web Dashboard供非技术用户可视化操作 |
| **费用模式** | 完全免费 | **基础工具免费+高级工具限免**——4个基础工具永久免费，订单/支付类高级工具企业版付费 |

**核心差异化定位**：瑞幸做的是"让AI帮你点咖啡"的消费入口，我们做的是**"让AI帮你管餐厅"的运营入口**——这是瑞幸完全未覆盖的场景，也是餐饮AI店长的核心价值主张。

### 2.5 肯德基案例

肯德基作为**首个接入千问的餐饮品牌** [(三湘都市报)](https://sxdsb.voc.com.cn/mobile/content/32820900)，具体做法如下：

| 维度 | 详情 |
|------|------|

| 会员权益打通 | 下单时自动使用大神卡及匹配的各类优惠券 [(新浪科技)](https://finance.sina.com.cn/tech/roll/2026-06-04/doc-iniafrtu9370097.shtml) |

3. **到店自取优先**：与瑞幸一样，配送暂不支持，优先跑通自取闭环

### 2.6 我们接入的可行性和优先级建议

#### 可行性评估

| 评估维度 | 评分(1-5) | 说明 |

#### 优先级建议

**优先级：P1（3周内启动）**

理由：
- 肯德基/瑞幸已验证餐饮Skill接入可行，技术路径清晰
- 千问用户基数大，服务类对话日超1亿次 [(中国经济新闻网)](https://www.cet.com.cn/itpd/hlw/10387480.shtml)
- 我们定位"餐饮管理"而非"餐饮消费"，与肯德基/瑞幸形成场景互补
- Agent形态仍在测试，建议先以Skill形态上线，抢占"餐饮AI店长"生态位

**接入策略**：
1. 第一阶段：封装4个MCP工具为千问Skill（菜单查询/库存状态/门店信息/日汇总）
2. 第二阶段：升级为Agent形态，增加主动提醒（库存预警、日汇总推送等）
3. 第三阶段：与千问会员体系打通，提供个性化服务

### 3.2 公众号/小程序接入AI能力的技术路径

#### 3.2.1 路径一：微信AI全局Agent接入（小程序被AI调用）

这是最核心的接入路径——让微信AI"小微"能理解用户需求并自动调用你的小程序完成服务。

**开发模式核心文件结构** [(掘金)](https://juejin.cn/post/7648857063246479386)：

```
小程序项目
├── app.json              # 声明Skill分包
├── skills/               # SKILL独立分包
│   └── order-skill/      # 点单Skill
│       ├── SKILL.md      # 业务说明（给AI看的SOP）
│       ├── mcp.json      # 接口声明（AI的"说明书"）
│       ├── index.js      # 接口注册
│       ├── apis/         # 原子接口实现
│       │   ├── getMenu.js
│       │   ├── searchProducts.js
│       │   ├── createOrder.js
│       │   └── ...
│       ├── components/   # 原子组件（卡片渲染）
│       │   ├── menu-card/
│       │   ├── order-card/
│       │   └── ...
│       └── utils/
└── page-meta.json        # 页面元数据
```

**核心概念** [(人人都是产品经理)](http://m.toutiao.com/group/7649677111069803043/)：

| 概念 | 说明 |
|------|------|
| **原子接口** | 最小执行单元，封装单一功能（如"查询菜单""创建订单"） |
| **原子组件** | 原子接口的可视化展示，渲染为对话流中的GUI卡片 |
| **SKILL.md** | 业务SOP，告诉AI流程编排规则（如"必须先选商品再下单"） |
| **mcp.json** | 声明可调用接口、参数、卡片映射，是AI的"能力说明书" |

**注意力权重表**（决定AI行为的关键优先级）[(人人都是产品经理)](http://m.toutiao.com/group/7649677111069803043/)：

| 信息来源 | 权重 | 说明 |
|---------|------|------|
| 原子接口返回的content | ★★★★★ | 离决策点最近，模型当作事实+动作读取 |
| mcp.json中的description | ★★★★ | 影响模型选不选这个接口 |
| inputSchema.description | ★★★★ | 影响模型怎么填参数 |
| SKILL.md | ★★★ | 适合写业务流程编排和跨接口规则 |

#### 3.2.2 路径二：小程序内置AI能力（extend.AI）

开通微信云开发后，页面/云函数直接调用混元/DeepSeek模型 [(头条)](http://m.toutiao.com/group/7651797185545142793/)：

```javascript
// 创建模型实例
const model = wx.cloud.extend.AI.createModel({
  model: 'hunyuan-lite' // 腾讯混元模型
});

// 流式对话
const stream = model.streamText({
  messages: [{ role: 'user', content: '推荐今日特价菜品' }]
});
```

适用场景：小程序内部AI对话、智能推荐、数据分析等，不需要全局Agent调度。

#### 3.2.3 路径三：iLink Bot API（智能体接入微信个人号）

- 注意：虽有法律条款背书，但需谨慎评估数据安全

#### 3.2.4 餐饮小程序推荐技术路径

### 3.3 微信AI Agent审核流程和要求

#### 3.3.1 审核核心维度 [(头条)](http://m.toutiao.com/group/7649579351952687631/)

| 维度 | 具体要求 |
|------|---------|
| **功能合规** | 技能描述与实际功能一致，无虚假宣传；餐饮类需补充食品经营许可证 |
| **数据安全** | 用户数据采集/存储/传输符合《网络安全法》；AI操作日志留存≥6个月 |
| **用户体验** | 自动模式技能操作链路清晰无卡顿；开发模式需覆盖异常场景（库存不足/支付失败） |
| **内容合规** | AI生成内容符合微信内容规范，无违法违规信息 |

#### 3.3.2 审核时效

| 模式 | 审核周期 | 说明 |
|------|---------|------|
| 自动模式 | 1-2个工作日 | 随小程序版本审核同步完成 |
| 开发模式 | 3-5个工作日 | 专项审核，含技术评测+安全审计+体验测试 |
| 驳回处理 | 无次数限制 | 明确驳回原因，修改后重新提交 |

#### 3.3.3 合规红线

- ❌ 禁止未经授权获取用户隐私数据
- ❌ 禁止AI操作跳转至外部违规链接
- ❌ 禁止金融类小程序无资质开展信贷/理财业务
- ❌ 禁止医疗类小程序提供诊断/处方开具服务
